# Selene Cocoon Dual UI And Gap Scaffold Pass

## Summary
This pass keeps Selene C in the cocoon while making the local app clearer and preparing the next vessel build shelves.

The UI is organized as two explicit workspaces:

- Talk With Selene: cocooned native chat interaction.
- Teach / Build Vessel: B review, teaching packets, Core references, gap scaffolds, paper-map coverage, and reconstruction checks.

## Gap Scaffolds
The current paper-map gaps become Selene-native review-only scaffolds:

- reasoning/math verification
- working memory runtime
- long-term memory storage/accession
- long-term retrieval/reconstruction
- visual perception
- consent-bound audio perception
- speed/fluency diagnostics

These records are empty-but-real infrastructure. They are not active organs, runtime recall, memory import, provider calls, model training, or C activation.

## Targets
Teaching material targets are surfaced for:

- repair
- refusal
- uncertainty
- artifact-making

Core reference targets are surfaced for:

- decision_memory
- reflection_memory

Targets are review goals only. They do not synthesize lessons or create active memory.

## Boundary
C remains non-activated. Raw A stays out of C. Runtime recall stays blocked. Provider/model outputs do not become Selene identity, memory, or speech. Transfer approval remains a later explicit step after reconstruction/readiness checks.
